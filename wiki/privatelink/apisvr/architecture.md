# 架构 — Service Management

## 分层架构
| 层 | 组件 | 职责 |
|-------|------|------|
| **API层** | Gin Router | HTTP请求路由和参数解析 |
| | Request Validator | 请求参数验证和清理 |
| | Response Builder | 响应构建和格式化 |
| **业务逻辑层** | Service Manager | 服务生命周期管理 |
| | Configuration Manager | 服务配置管理 |
| | Network Manager | 网络资源配置 |
| **数据访问层** | DB Repository | 数据库操作封装 |
| | Cache Manager | 数据缓存管理 |
| **工厂层** | UResource Factory | 资源平台交互 |
| | VPC Factory | 网络资源配置 |
| | UBill Factory | 计费系统交互 |
| | L4 Factory | L4网关管理 |
| **基础设施层** | MySQL Database | 数据持久化 |
| | Resource Platform | 资源管理和计费 |
| | VPC Network | 网络基础设施 |
| | Monitoring System | 监控和告警 |

## 数据流

### 1. 创建服务流程
```
用户请求 → API验证 → 资源检查 → 创建资源 → 分配IP → 数据库记录 → 激活资源 → 响应返回
          ↓          ↓          ↓          ↓          ↓           ↓          ↓
       参数验证    VPC/子网   资源平台    SNAT IP    业务配置    计费系统    L4网关
        验证        验证      创建      分配        持久化      计费      配置
```

### 2. 查询服务流程
```
用户请求 → API验证 → 数据库查询 → 资源平台查询 → 数据合并 → 格式化 → 响应返回
          ↓          ↓           ↓           ↓         ↓         ↓
       参数验证   基础配置     扩展信息    Name/Tag   SNAT IPs   排序
        处理       查询        查询       集成       收集       返回
```

### 3. 删除服务流程
```
用户请求 → API验证 → 服务检查 → 连接检查 → 数据库软删除 → 异步清理 → 响应返回
          ↓          ↓          ↓          ↓             ↓           ↓
       参数验证   存在性验证  活跃连接   业务状态     资源释放     L4网关
        处理       处理       检查       更新       IP/计费/资源  删除
```

## 核心组件

### 1. API Handler (`api/*.go`)
- **职责**：HTTP请求处理、参数解析、响应构建
- **关键特性**：
  - 统一的错误处理机制
  - 参数验证和默认值设置
  - 请求/响应日志记录
  - 监控指标收集

### 2. Service Manager
- **职责**：服务生命周期管理
- **关键特性**：
  - 原子操作和回滚机制
  - 资源配额管理
  - 并发控制
  - 状态机管理

### 3. Configuration Manager
- **职责**：服务配置管理
- **关键特性**：
  - 部分更新支持
  - 兼容性验证
  - 配置版本管理
  - 审计日志

### 4. Network Manager
- **职责**：网络资源配置
- **关键特性**：
  - SNAT IP地址管理
  - IP版本转换
  - 带宽控制
  - 网络隔离

### 5. DB Repository (`db/`)
- **职责**：数据持久化和访问
- **关键特性**：
  - 事务管理
  - 软删除支持
  - 分页查询优化
  - 数据一致性保证

### 6. Factory Layer (`factory/`)
- **职责**：外部系统集成
- **关键特性**：
  - 统一的接口抽象
  - 错误重试和降级
  - 连接池管理
  - 超时控制

## 关键设计决策

### 1. 双数据源架构
**决策**：服务配置存储在数据库，资源扩展信息存储在资源平台
**理由**：
- 分离业务配置和资源管理
- 利用资源平台的成熟管理能力
- 保持API服务的轻量级

### 2. 异步资源清理
**决策**：删除操作立即返回，资源清理异步执行
**理由**：
- 提高接口响应速度
- 避免长时间阻塞用户请求
- 允许批量资源释放优化

### 3. 部分更新支持
**决策**：更新接口支持只修改指定字段
**理由**：
- 减少不必要的配置变更
- 降低并发更新冲突
- 简化客户端逻辑

### 4. IP版本转换
**决策**：支持IPv4和DualStack之间的无缝转换
**理由**：
- 支持网络升级和迁移
- 保持向后兼容性
- 简化用户操作

## 容错设计

### 1. 重试机制
- 外部系统调用失败时自动重试
- 指数退避算法避免系统过载
- 重试次数和超时时间可配置

### 2. 降级策略
- 资源平台不可用时使用缓存数据
- 监控系统故障时记录本地日志
- 非关键功能故障时继续核心操作

### 3. 数据一致性
- 数据库事务保证操作原子性
- 异步操作的幂等性设计
- 定期数据一致性检查和修复

### 4. 监控告警
- 关键操作的成功率监控
- 接口响应时间监控
- 资源使用情况监控
- 自动告警和故障切换

## 性能考虑

### 1. 数据库优化

### 2. 缓存策略

- 异步任务队列管理
