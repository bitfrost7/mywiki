从数据来源分，知识库分为 1，内部和外部，内部 比如 企业内部知识库，内部代码库总结，个人学习总结 外部 比如 公众号文章 博客文章 官方文档 github 开源仓库总结。从来源置信度分为 源码型 和 文档型， 源码型指 有编译通过 测试通过的功能完善的代码，如内部项目仓库的 master 分支， 开源项目的 master 分支 等。文档型指 内部文档，公众号文章，由人总结编写 ，置信度不可靠

知识处理流程:

代码型:

1. 初步编译: 使用codegraph或者其他graphy 工具，对各个仓库生成对应的 ast骨架。
2. 对仓库粒度的 repo 生成 文档，包含：
    a. readme，包含整体项目的功能介绍，整体架构简要介绍，各模块的简要介绍。目的是回答问题 这是一个什么项目 做什么的
    b. architecture 整体架构介绍，项目分层，每层各模块的作用，关系。 目的是回答问题，这个项目的架构是什么，分几层 都分别是做什么的
    c. modules/module.md 主要是分模块去介绍 各个模块内部自己的 功能，输入输出是什么。
    d. dataflow.md 主要是介绍数据流转的过程，目的是回答 用户请求/流量数据来了之后这个服务是怎么处理的, 如果具体的数据流比较多，目录下分文件
        I: dataflow/{api接口}.md: api/grpc 服务, 主要介绍 该接口内用户请求到各个模块的流转，比如 apisvr 创建 endpoint，分参数解析层，会进行参数校验， 接着到业务规则校验层，这里涉及哪几个模块，接着到内部处理层，这里涉及哪几个模块，最后到 dao 层，主要是对哪几个库进行操作
        II: dataflow/{dataflow数据路径}.md: 如果是 单纯的数据处理，类似网络流量或者业务数据流转等。思路和 api 一样，也是介绍 例如 流量的入口 解析层 都有哪些模块分别做什么，流量处理层 哪些模块 都是做什么 流量转发层 哪些模块 都是做什么，区别仅是 没有具体的 api 接口而是以路径命名
        III: 以上都是举例不要死板代入，核心是描述 流入请求 分层，分模块之间的流转关系。 还有如果说有那种 既有 api 也有 dataflow 的 需要都描述。
    e. 注意需要有具体的代码来源，方便后续溯源
    这一步采用 hermes kanban ，通过简单的 workflow.yaml 描述 ，去运行 kanban。 
    orchestrator 负责任务调度
    编写: 2-4 个 writter 并行根据 ast图，源码，项目readme，去按照orchestrator 分配的任务 和 prompt生成文档。
    review: reviewer按照评判标准对writter内容进行挑刺，事实来源是源码和ast图，review后的意见继续加入 prompt 去让 writter 修改，直到没有意见
    润色： 在不改变逻辑的情况下，对文档内容进行简单的排序，加入不改变语义的词语使得文档更易读。因为writter设定必须百分百遵守原逻辑，可能生成的文档比较生硬。

3. 对业务粒度按照服务的文档生成提炼文档，包含:
    a. architecture.md, 整个业务的架构，目的是 能回答服务间的关系是什么 
    b. dataflow.md, 整体业务的各个服务之间的数据流转，目的是回答当来了一个请求，流转路径是什么样的

    这一步是对第 2 步的文档的一个总结和提炼，但依然也要有对应的代码事实支撑

    kanban 流程
    编写: 架构师 profile 去阅读各个服务的文档，提炼出architecture.md和dataflow.md
    reviewer: 根据代码事实对architecture.md，dataflow.md进行挑刺，依旧是直到改到没有review意见
    润色: 在不改变逻辑的情况下, 对文档进行润色

文档型：

1. 初步编译： 对文档自身逻辑/语法进行校验，找出自身逻辑/语法不合理的地方，进行汇总报告 请求人类去确认 状态 raw->verified
2. 文档交叉编译：对 1 步骤过了的 文档，和已经有的知识库进行逻辑校验，防止出现逻辑冲突 状态 verified-merge
3. 文档归并， agent对文档进行语义理解，给出合并到知识库的意见，提出审批，状态 merge-> approve


文档型 直接应该就是一个skill，用户对话中提出 对某文档进行入库 ，运行上述流程。

wiki编译
分为 personal，个人文档，biz， 公司项目和公司业务文档  public，外源性 公众号 github提炼出的知识
wiki的内容 需要包含引用来源的文档

置信度机制：
1-10
代码型知识和文档型知识 都有一个基础的置信度 代码型较高 7-8 文档型因为有 review机制 8-9
置信度本身也有一个时间衰减机制每周衰减1

置信度扣减 主要是文档被命中后 来自用户的反馈，用户点赞会提高置信度。 

文档的知识库置信度低于阈值会 发起修复流程， 这时候会起一个fix流程

1. fixer根据文档内容，反馈意见和已有高于等于8的文档，如果文档是来源于代码事实，还会去更新代码仓库。fixer会给出修改意见
2. 人类介入审批修改意见，通过后fixer 进行修复，修复后的文档会让reviewer介入去和事实进行对比，如果不符合打回
