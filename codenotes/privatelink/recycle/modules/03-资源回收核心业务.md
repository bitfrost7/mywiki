# 资源回收核心业务 — Recycle Business Logic

## 1. 模块职责

本模块是 recycle 服务的**核心业务逻辑层**，社区 C9 / C32 / C33 共 3 个源文件，负责 VPC Endpoint 和 Endpoint Service 的**三阶段回收生命周期管理**：

- **停服（Close）**：欠费后标记资源为关闭状态，断开所有连接（`close_status=1`，`end_time=now`）
- **恢复（Recover）**：停服后可恢复，更新关闭状态并重建连接记录（`close_status=0`，`INSERT start_time`）
- **删除（Delete）**：超期后调用 PrivateLink API 彻底删除资源并通知回收系统
- **外部调用编排**：调用 URecycleV2（获取列表/通知）→ UResource（查询 RegionId）→ PrivateLink（执行删除）
- **数据库事务**：通过 GORM `Transaction` 确保状态更新的一致性

### 3.6 延迟停服机制

延迟校验条件（`task/recycle_endpoint.go:29`, `task/recycle_service.go:30`）：

```go
if time.Now().Unix() - int64(resource.CloseTime) > t.delay {
    t.CloseEndpoint(ctx, resourceInfo)
}
```

- `t.delay` 单位为秒，默认 5 分钟 = 300 秒（`NewServer` 传入 `cfg.CloseDelayTime*60`，`server.go:134`）
- `CloseTime` 由 URecycleV2 系统设置，表示预期停服时间戳
- 作用：防止欠费后立即停服，给用户充值恢复的缓冲时间

### 3.7 错误处理策略

| 失败场景 | 行为 | 指标 |
|----------|------|------|
| URecycle 获取列表失败 | 记录 ERROR 日志，直接 return | 无 |
| UResource 查询失败 | 记录 ERROR 日志，`continue` 跳过本条 | 无 |
| DB 操作失败 | 记录 ERROR 日志，`RecycleFailCounter.Inc(type)` | `type="close"` / `"recover"` |
| PrivateLink 删除失败 | 记录 ERROR 日志，`RecycleFailCounter.Inc("delete")` | `type="delete"` |
| 通知 URecycle 失败 | 记录 ERROR 日志，`RecycleFailCounter.Inc(type)` | 同 type |

**注意**：DB 操作失败后立即返回（`task/recycle_endpoint.go:43`），通知操作不会执行。
