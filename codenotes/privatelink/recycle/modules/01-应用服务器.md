## 3. 关键实现逻辑

### 3.1 服务启动流程（`NewServer` @ `server.go:77-116`）

```
1. s.Init(cfg)                     — app 框架初始化（日志、ZK 客户端等）
2. cfg.SetDefaultValue()           — 配置默认值填充
3. ZK.NewMaster2()                 — ZooKeeper Master 选举注册 (L85-90)
4. db.NewDatabase()                — GORM + MySQL 连接 + 监控插件 (L93-96)
5. factory.InitFactory()           — HTTP Client 初始化 (L99-100)
6. InitCronTask()                  — 注册 4 个定时任务 (L102-105)
7. InitApi()                       — 初始化 Gin API 路由 (L107-108)
8. CollectSysMetrics()             — 启动系统指标协程 (L111-113)
```

### 3.2 Master 选举机制

recycle 使用 ZooKeeper 实现 Master 选举（`server.go:83-88`）：

- `ZK.NewMaster2()` 在 ZK 路径上注册临时节点，抢注成功的节点成为 Master
- Master 状态通过 `atomic` 包的 `uint32` 标志位维护（`server.go:72`）
- `MasterLoop`（`server.go:185-203`）：CAS 切换状态（0→1 升主，1→0 降备），阻塞监听 ZK 上下文
- 每个 cron 任务执行前调用 `isMaster()`（`task/task.go:31-37`）：非 Master 节点跳过任务执行
- API 请求前调用 `isMaster()`（`server.go:123-127`）：非 Master 节点返回错误
- `monMaster`（`server.go:209-215`）：每 5 秒将主从状态上报到 `privatelink_recycle_common_gauge{type="is_master"}`

### 3.3 Cron 任务注册（`InitCronTask` @ `server.go:132-175`）

使用 `cron.WithSeconds()` 支持秒级精度。注册 4 个定时任务，每个任务内部处理 Endpoint 和 Service 两类资源：

| 序号 | 任务 | 回调（两行代码） | cron spec 来源 |
|------|------|-----------------|---------------|
| 1 | 停服 | `CloseEndpointWithDelay` + `CloseEndpointServiceWithDelay` | `cfg.ToCloseSpec`（`server.go:137-139`） |
| 2 | 恢复 | `RecoverEndpoint` + `RecoverEndpointService` | `cfg.ToRecoverSpec`（`server.go:147-149`） |
| 3 | 删除 | `DeleteEndpoint` + `DeleteEndpointService` | `cfg.ToDeleteSpec`（`server.go:157-159`） |
| 4 | 主从监控 | `monMaster` | `cfg.MonMasterSpec`（`server.go:167-169`，典型 `*/5 * * * * *`） |

**Cron 执行约束**（`task/task.go:31-37`）：
```go
c.AddFunc(spec, func() {
    if isMaster != nil {
        if isMaster() { /* 执行任务 */ } else { return }
    }
    if run != nil { run() }
})
```
