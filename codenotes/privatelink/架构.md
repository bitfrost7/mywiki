## 服务职责总览

### API 网关层 (HTTP Action 路由)

#### 1. apisvr — PrivateLink API Server

**定位**：系统主 API 网关，对外暴露所有用户操作接口。单端点 `POST /`，Action 字段分发。

**核心职责**（`apisvr/api/` 目录下各 Handler）：

| 功能域 | Action 示例 | 源码位置 |
|--------|------------|----------|
| VPC Endpoint Service CRUD | `CreateVPCEndpointServiceConfiguration`, `DeleteVPCEndpointServiceConfiguration`, `DescribeVPCEndpointServiceConfigurations`, `UpdateVPCEndpointServiceConfiguration` | `api/CreateVPCEndpointServiceConfiguration.go:128` 等 |
| VPC Endpoint CRUD | `CreateVPCEndpoint`, `DeleteVPCEndpoint`, `DescribeVPCEndpoints`, `UpdateVPCEndpointAttribute` | `db/db.go:250`, `api/DeleteVPCEndpoint.go:21` |
| Connection 管理 | `AcceptVPCEndpointConnection`, `RejectVPCEndpointConnection`, `DescribeVPCEndpointConnections`, `UpdateVPCEndpointConnectionAttribute` | `api/AcceptVPCEndpointConnection.go:30` 等 |
| 用户权限管理 | `AddUsersToVPCEndpointService`, `RemoveUsersToVPCEndpointService`, `UpdateUsersToVPCEndpointService`, `ListVPCEndpointServiceUsers` | `api/AddUsersToVPCEndpointService.go:37` 等 |
| 价格/带宽查询 | `GetPrivateLinkPrice`, `GetPrivateLinkBandwidth` | `api/GetPrivatelinkPrice.go:41`, `api/GetPrivateLinkBandwidth.go:21` |

**外部调用**（`server.go:79-85`）：
- **VPC** (`UVPCFEGO`) — 分配 IP、查询 VPC 网络
- **LB** — 负载均衡器操作
- **L4** (`privatelink-l4fe` 经 ZK 发现) — 创建/删除 L4 网关、获取 VIP
- **UResource** — 资源标签管理
- **UAccount** — 账户信息查询
- **UBill** — 价格查询

#### 2. privatelink-ops — 运维 API 服务

**定位**：运维操作入口，提供资源可见性控制（设置/取消不可见标签）和用户配置 CRUD。

**核心职责**（`privatelink-ops/api/` 目录）：

| Action | 说明 | 源码位置 |
|--------|------|----------|
| `SetResourceInvisible` | 设置资源不可见（更新 DB + UResource 标签） | `privatelink-ops/architecture.md:99-129` |
| `SetResourceVisible` | 取消资源不可见标签 | `privatelink-ops/architecture.md:101-128` |
| `AddUserConfig` | 新增用户配置项 | `privatelink-ops/api/AddUserConfig.go:29` |
| `DeleteUserConfig` | 删除用户配置项 | `privatelink-ops/api/DeleteUserConfig.go:24` |
| `UpdateUserConfig` | 更新用户配置值 | `privatelink-ops/api/UpdateUserConfig.go:25` |

**外部调用**：UResource（设置/删除资源标签）

#### 3. recycle (HTTP API 面) — 带宽回收 API

recycle 同时提供 HTTP API 和 Cron 任务（见下文）。API 面仅处理 **VPC Endpoint** 的操作。

| Action | 说明 | 源码位置 |
|--------|------|----------|
| `CloseEndpoint` | 触发批量停服扫描 | `recycle/api/recycle.go:19` |
| `RecycleEndpoint` | 触发批量恢复扫描（路由映射到 RecoverEndpoint） | `recycle/api/recycle.go:30` |
| `DeleteEndpoint` | 触发批量删除扫描 | `recycle/api/recycle.go:41` |

### 业务服务层 (Cron 驱动 / gRPC)

#### 4. billing — 定时计费服务

**定位**：纯后台 cron 驱动，无 HTTP API。通过 ZooKeeper 选主确保单 master 执行计费。

**5 个定时任务**（`server.go:116-163`）：

| 任务 | 周期 | 说明 |
|------|------|------|
| TrafficBilling | 每小时整点 | 读取 `t_traffic_info` 计算流量费，调用 UBill `PostPaid` 扣费 |
| TrafficReBilling | 按需重试 | 重试失败流量订单，先查 UBill 是否已支付 |
| InstanceBilling | 每小时整点 | 读取连接信息计算实例费，调用 UBill `PostPaid` 扣费 |
| InstanceReBilling | 按需重试 | 重试失败实例订单 |
| MonMaster | 每 5 秒 | 上报主从状态到 Prometheus |

**外部调用**：UBill (`PostPaid`, `GetPostPaidItemBillInfo`, `GetUnpaidOrderDetailInfo`)

#### 5. billinsert — 流量数据插入服务

**定位**：纯后台 cron 驱动，每小时拉取 CloudWatch 监控数据，计算梯形积分法流量，写入 `t_traffic_info`。

**2 个定时任务**（`billinsert/architecture.md:54-74`）：

| 任务 | 周期 | 说明 |
|------|------|------|
| InsertTrafficInfo | 每小时第 2 分钟 | 拉取 CloudWatch 指标 → 梯形积分计算 → 批量写入 |
| MonMaster | 每 5 秒 | 上报主从状态 |

**外部调用**：CloudWatch Admin API（拉取带宽监控数据）

#### 6. bwregister — 带宽注册同步服务

**定位**：Cron 驱动 + gRPC 双向。作为 gRPC Server 对外提供限速信息查询，作为 gRPC Client 同步到 UTraffic 平台。

**4 个定时任务**（`bwregister/architecture.md:78-82`）：

| 任务 | 周期 | Master 限制 | 说明 |
|------|------|-------------|------|
| SyncAllConnectionInfo | 10 秒 | ✅ | 全量同步流量信息到 UTraffic |
| SyncUserConfig | 60 秒 | ❌ | 同步用户配置到内存 |
| monMaster | 5 秒 | ❌ | 上报主从状态 |
| RegisterTrafficManagerInfo | 10 秒 | ✅ | 注册节点到 ZK 路由 |

**gRPC Server 接口**（`api/grpc.go:24,40`）：
- `GetAllTrafficInfo` — 返回全量资源限速信息
- `GetAllShareBWInfo` — 返回全量共享带宽信息

**gRPC Client 调用**（`task/limiter.go:170,190,210`）：
- `SyncAllTrafficInfo` → UTraffic 平台
- `SyncAllShareBWInfo` → UTraffic 平台
- `RegisterTrafficManagerInfo` → UTraffic 平台

#### 7. privatelink-l4fe — L4 转发引擎

**定位**：轻量级 gRPC 数据服务，为 apisvr 提供 L4 Gateway 配置查询。

**gRPC 接口**（`api/grpc_api.go:26`）：
- `ListL4Gw(ListL4GwRequest) → ListL4GwReply` — 批量查询 L4 Gateway 配置

**FullNAT 模型**（`privatelink-l4fe/architecture.md:98-103`）：
- Fore (前端) = VPC Endpoint 的 IP
- Backend (后端) = Service 自身的 IP
- FNAT (SNAT) = Service SNAT IP
- Rules = FullNAT 转发规则

#### 8. recycle (Cron 面) — 带宽回收服务

**定位**：Cron 驱动 + HTTP API 双通道。三阶段回收模型。

**4 个定时任务**（`recycle/server.go:132-175`）：

| 任务 | 说明 | 处理资源 |
|------|------|----------|
| ToClose | 延时停服 — 查待停服列表 → 过滤 RegionId → 校验时间 → 停服 | Endpoint + Service |
| ToRecover | 恢复窗口 — 查可恢复列表 → 更新状态 → 重建连接 | Endpoint + Service |
| ToDelete | 删除回收 — 查待删列表 → PrivateLink 删除 → 通知系统 | Endpoint + Service |
| MonMaster | 每 5 秒上报主从状态 | — |

**三阶段回收模型**（`recycle/architecture.md:23-29`）：
```
欠费 → [停服(CloseDelayTime缓冲)] → [可恢复窗口(24h)] → [删除(不可恢复)]
```

**外部调用**：
- **URecycleV2** — 获取待处理列表 + 通知状态变更
- **UResource** — 查询资源 RegionId
- **PrivateLink** (apisvr 的 HTTP API) — 执行 `DeleteVPCEndpoint` / `DeleteVPCEndpointService`

## 关键设计模式

### 1. ZooKeeper 主从选举

5 个服务使用 ZooKeeper `NewMaster2()` 实现 Master 选举（`billing/server.go:83-88`, `billinsert/server.go:83-88`, `bwregister/server.go:76`, `recycle/server.go:83-88`）：

```
启动流程:
1. ZK.NewMaster2(servicePath, lockPath)   — 注册竞选
2. MasterLoop()                            — 监听 ZK 上下文
3. isMaster() → atomic.CompareAndSwapUint32 — CAS 检查
4. cron 任务/API 前检查 isMaster()          — 仅 Master 执行
```

**使用 Master 选主的服务**：
- **billing**: 计费任务（仅 Master 扣费，防止重复扣费）
- **billinsert**: 流量插入（仅 Master 写入，防止重复）
- **bwregister**: 带宽同步（仅 Master 同步到 UTraffic）
- **recycle**: 回收任务 + HTTP API（仅 Master 回收）
- **apisvr**: ⚠️ 未使用 ZK 选主（无状态 API 网关，多副本均可服务）
- **privatelink-ops**: ⚠️ 未使用 ZK 选主

**不使用的服务**：
- **privatelink-l4fe**: 无状态 gRPC 查询服务，未使用选主
- **privatelink-utils**: 工具库，不部署

### 2. Cron 驱动 vs API 驱动

| 模式 | 服务 | 说明 |
|------|------|------|
| **纯 Cron 驱动** | billing, billinsert | 无 HTTP API 入口，全部行为由定时任务驱动 |
| **Cron + HTTP API** | recycle, bwregister | Cron 执行批量操作，HTTP/gRPC 接收外部请求或触发立即执行 |
| **纯 API 驱动** | apisvr, privatelink-ops, privatelink-l4fe | 响应外部请求，无定时任务 |

### 3. Action 字段路由 (JSON RPC 风格)

所有 HTTP 服务（apisvr, privatelink-ops, recycle）统一使用 `POST /` + JSON 中的 `Action` 字段进行分发。共享基类 `ReqBase`/`RespBase` 模式（`privatelink-ops/api/base.go:8-17`, `recycle/api/api.go:32-42`）。

### 4. 共享数据库 + 读写分离

全部服务连接同一个 MySQL 实例（各服务独立配置 DSN），通过 GORM Gen 框架自动生成 model/query 代码。GORM 支持读写分离（`db/query/gen.go:ReadDB()/WriteDB()`），大部分查询使用 `Use()` 统一入口。

### 5. 三层 Factory 封装（调用外部系统的标准模式）

每个对外部系统的 HTTP 调用都遵循 `impl.go → basic.go → expand.go` 三层结构（`recycle/architecture.md:201`）：
- `impl.go` — 客户端初始化，持有 `httpclient.HTTPClient`
- `basic.go` — 请求/响应结构体定义，单次 API 调用方法
- `expand.go` — 批量/组合逻辑，重试与错误处理

### 6. Prometheus 全覆盖监控

每个服务均埋点上报：请求计数/延迟、DB 操作耗时、任务执行状态、系统资源（CPU/内存/DB连接池）。通过 `privatelink-utils/gorm/plugin/prometheus.go` 的 GORM 插件统一采集数据库指标。
