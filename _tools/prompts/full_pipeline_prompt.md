# 代码知识库文档生成任务

## 任务目标

为 `privatelink/apisvr` 代码仓库生成人类可读、层次清晰的知识库文档。

## 目标仓库

- **系统**: privatelink（私有连接服务）
- **仓库**: apisvr
- **路径**: `~/.cache/mywiki-repos/apisvr` 或 `~/Documents/Code/work/privatelink/apisvr`
- **语言**: Go

## 输出要求

生成以下 5 类文档，每类都是独立的 Markdown 文件：

### 1. 项目概览 (README.md)

**内容**：
- 项目名称和一句话描述
- 核心职责（2-3 点）
- 技术栈（语言、框架、关键依赖）
- 目录结构简述

**格式示例**：
```markdown
# apisvr

> PrivateLink API 服务，处理终端用户的连接请求管理

## 核心职责

1. 接收并验证用户连接配置
2. 协调后端资源分配
3. 提供连接状态查询接口

## 技术栈

| 类别 | 技术 |
|------|------|
| 语言 | Go 1.22 |
| 框架 | Gin |
| 存储 | MySQL, Redis |
| ... | ... |

## 目录结构

- `internal/api/` — HTTP 接口处理
- `internal/service/` — 业务逻辑
- ...
```

**约束**：总字数控制在 300 字以内，只描述已观察到的代码。

---

### 2. 架构设计 (architecture.md)

**内容**：
- 架构模式（分层架构/微服务/DDD等）
- 核心模块划分及职责
- 关键接口/抽象定义
- 数据流概览

**格式要求**：
```markdown
## 架构模式

本项目采用 [模式名] 模式...

## 模块划分

| 模块名 | 主要文件 | 职责 | 对外接口 |
|--------|----------|------|----------|
| auth | `internal/auth/handler.go` | 身份验证 | `AuthMiddleware()` |
| ... | ... | ... | ... |

## 关键抽象

```go
// internal/common/interface.go:23
type ConnectionManager interface {
    // ...
}
```

## 数据流

典型请求处理流程：
1. HTTP 请求 → Handler 层
2. 参数验证 → Service 层
3. 业务逻辑 → DAO 层
4. 数据库操作 → 返回响应
```

**约束**：
- 使用 Markdown 三级标题
- 每个模块标注**代码来源**，如 `internal/api/handler.go:42-56`
- 优先展示 interface/struct 定义，而非函数实现细节

---

### 3. API 文档 (api.md)

**内容**：
- 接口清单表格
- 关键接口详情
- 路由注册信息

**格式要求**：
```markdown
## 接口清单

| 路由 | 方法 | 处理函数 | 文件位置 | 功能简述 |
|------|------|----------|----------|----------|
| /api/v1/users | GET | GetUsers | handler.go:45 | 获取用户列表 |
| ... | ... | ... | ... | ... |

## 关键接口详情

### POST /api/v1/users

**功能**: 创建新用户

**请求参数**:
| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| name | string | 是 | 用户名 |
| email | string | 是 | 邮箱 |

**处理逻辑**:
1. 验证参数格式
2. 检查邮箱是否已存在
3. 写入数据库
4. 返回用户 ID

**代码位置**: `internal/api/user_handler.go:67-89`

## 路由注册

```go
// internal/router/router.go:23
r.Group("/api/v1")
  ├── GET  /users      → userHandler.List
  ├── POST /users      → userHandler.Create
  └── ...
```
```

**约束**：
- 只展示能从代码确认的接口
- 每个接口必须标注**代码位置**（文件 + 行号）

## 工作流程建议

1. **先浏览整体结构** — 查看 `go.mod`、`main.go`、目录结构
2. **提取关键代码** — 识别 api/db/config/integration 类型的代码
3. **分层生成** — 先概览，再架构，最后模块详情
4. **标注来源** — 每个重要信息都标注代码位置
